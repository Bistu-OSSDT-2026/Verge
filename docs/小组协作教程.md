## 1. 基础概念

### Git 是什么？

Git 是一个**版本控制工具**，记录你对项目做的每一次修改。相当于游戏的"存档系统"——你可以随时回到之前的任何一个版本。

### 三个核心概念

| 概念 | 类比 | 说明 |
|------|------|------|
| **仓库 (Repository)** | 游戏的安装目录 | 存放所有代码和资源的地方 |
| **提交 (Commit)** | 游戏的存档点 | 记录一次完整的修改，"我改了啥" |
| **分支 (Branch)** | 游戏的"平行世界" | 在独立的世界里修改，不影响主世界 |

### 工作流程图示

```
你本地的电脑                         GitHub 云端
┌─────────────┐                    ┌─────────────────┐
│ 1. 克隆项目  │  ←──── 下载 ────  │  项目仓库        │
│ 2. 创建分支  │                    │  (大家的代码)    │
│ 3. 写代码    │                    │                 │
│ 4. 提交修改  │                    │                 │
│ 5. 推送到云端 │  ──── 上传 ────→  │ 6. 创建 PR      │
│             │                    │ 7. Review + 合并 │
└─────────────┘                    └─────────────────┘
```

### Pull Request 是什么？

PR 就是**"我改好了，请看看能不能合并到主项目"的申请单**。

类比：你写了一章小说，提交给编辑审核，编辑说"这里要改"，你改完后再提交，编辑说"OK"，然后合并到正式出版物。

## 2. 克隆项目

### 第一步：安装 Git

1. 下载 Git：https://git-scm.com/download/win
2. 安装时一路点"下一步"即可（用默认选项）
3. 安装完成后，在桌面右键菜单里会出现 "Open Git Bash here"

### 第二步：克隆项目到你的电脑

打开 Git Bash（或 PowerShell），输入：

```bash
# 进入你想存放项目的目录，比如桌面
cd ~/Desktop

# 克隆项目（下载整个项目到本地）
git clone https://github.com/Bistu-OSSDT-2026/Verge.git

# 进入项目目录
cd Verge
```

现在你电脑上 `~/Desktop/Verge/` 就是项目的完整副本。

### 第三步：用 Godot 打开项目

1. 打开 Godot Engine
2. 点击"导入"→ 选择 `~/Desktop/Verge/project.godot`
3. 开始开发！

## 3. 日常开发

### 每次开发的标准流程

```bash
# 1. 切换到 master 分支
git checkout master

# 2. 拉取队友最新的代码（重要！每次开始工作前都要做）
git pull

# 3. 创建你的开发分支（用你的名字 + 功能名）
git checkout -b feature/你的名字-功能名

# 4. 在 Godot 中写代码、改场景...

# 5. 查看你改了什么
git status

# 6. 把改动加入暂存区
git add -A

# 7. 提交（写清楚你做了什么）
git commit -m "修复了胜利结算面板按钮无法点击的问题"

# 8. 推送到 GitHub
git push -u origin feature/你的名字-功能名
```

### 提交信息规范

好的提交信息：
```
✅ "修复胜利结算面板返回章节按钮点击无效"
✅ "调整敌人数值：杂兵 180→100 HP"
✅ "添加 GitHub Actions 自动导出工作流"
```

不好的提交信息：
```
❌ "改了一些东西"
❌ "修bug"
❌ "update"
```

## 4. 分支操作

### 为什么要用分支？

**想象这个场景**：你和队友同时修改 `main_game.tscn`。你在上面加了一个新角色，他在上面改了一个 UI 按钮。

- **不用分支**：你们互相覆盖对方的代码，炸了。
- **用分支**：你在 `feature/新角色` 分支上改，他在 `fix/UI按钮` 分支上改。互不影响，最后各自提 PR 合并。

### 分支命名规范

```
feature/功能描述    →  新功能
fix/问题描述        →  修 Bug
docs/文档描述       →  文档修改
```

示例：
```
feature/敌人阻挡逻辑
fix/暂停按钮点不了
docs/更新协作教程
```

### 常用分支命令

```bash
# 查看所有分支
git branch -a

# 创建并切换到新分支
git checkout -b feature/你的功能名

# 切换到已有分支
git checkout master

# 删除本地分支
git branch -d feature/已完成的功能名
```

## 5. Pull Request

### 完整 PR 流程

#### Step 1: 确保你的分支已推送到 GitHub

```bash
git push -u origin feature/你的功能名
```

#### Step 2: 在 GitHub 网页上创建 PR

1. 打开项目页面：https://github.com/Bistu-OSSDT-2026/Verge
2. 点击 **Pull requests** 标签
3. 点击绿色 **New pull request** 按钮
4. 在 `compare` 下拉菜单选择你的分支
5. 点击 **Create pull request**
6. 填写 PR 标题和描述

#### PR 描述模板

```markdown
## 做了什么
- 修复了胜利结算面板按钮无法点击的问题
- 把暂停机制从 Engine.time_scale 改为 get_tree().paused

## 如何验证
1. 进入教学关
2. 按 F3 触发胜利结算
3. 点击"返回章节列表"按钮 → 应该能正常跳转

## 改动文件
- scripts/ui/victory_panel.gd
- scripts/ui/pause_menu.gd

## 截图（如有 UI 改动）
[粘贴截图]
```

#### Step 3: 等待 Review

队友会在 PR 页面留言：
- ✅ **Approve**（通过）：你的代码没问题，可以合并
- 💬 **Comment**（建议）：有小问题需要讨论
- ❌ **Request changes**（要求修改）：有必须修复的问题

#### Step 4: 根据意见修改

队友说"这里要改"，你在本地：

```bash
# 切换到你的分支
git checkout feature/你的功能名

# 修改代码...

# 提交并推送
git add -A
git commit -m "根据 Review 意见修改：修复了 XXX 问题"
git push
```

推送到 GitHub 后，PR 会自动更新，队友可以看到你的新改动。

#### Step 5: 合并

Review 通过后，点 **Merge pull request** → 你的代码就正式进入主项目了！

## 6. 代码 Review

### 作为 Reviewer（审查者）怎么做

1. 打开队友的 PR
2. 点击 **Files changed** 标签，查看所有改动
3. 在具体代码行上点击 **+** 号添加评论
4. 写清楚你的意见

### Review 应该写什么

```
✅ 好的 Review：
"这里的 get_tree().paused 改得好，解决了按钮点击问题。但建议
在 _close() 里也加上 Engine.time_scale 的恢复，防止倍速状态残留。"

✅ 好的 Review：
"第 45 行的变量名 level_start_time 和 第 67 行的 levelStartTime
风格不统一，建议统一用 snake_case。"

❌ 无效的 Review：
"LGTM"（课程文档明确说这个不算有效 Review）
"好的"
"没问题"
```

## 7. 解决冲突

### 什么时候会冲突？

你改了 `character_base.gd` 的第 10-20 行，队友也改了同一文件的同一区域，Git 不知道用谁的版本。

### 如何解决

```bash
# 1. 切换到 master 并拉取最新代码
git checkout master
git pull

# 2. 切回你的分支
git checkout feature/你的功能名

# 3. 把 master 的最新代码合并到你的分支
git merge master

# 如果提示冲突：
# Auto-merging scripts/character_base.gd
# CONFLICT (content): Merge conflict in scripts/character_base.gd
```

4. 打开冲突文件，会看到：
```
<<<<<<< HEAD
你的代码
=======
队友的代码
>>>>>>> master
```

5. 手动决定保留哪部分（或者两段都保留），删掉 `<<<<` `====` `>>>>` 标记
6. 保存文件，然后：
```bash
git add -A
git commit -m "解决合并冲突"
git push
```

## 8. GitHub Issues

### Issue 是什么？

Issue 是任务卡片，用来追踪"谁在做什么，做到哪了"。

### 创建 Issue

1. 进入项目 → **Issues** 标签
2. 点 **New Issue**
3. 写清楚：
   - 要做什么
   - 为什么需要做
   - 完成标准是什么

### Issue 模板

```markdown
### 问题描述
胜利结算面板的"返回章节列表"按钮点击无响应

### 期望行为
点击按钮后跳转到章节选择页面

### 当前行为
点击按钮无任何反应

### 负责人
@你的队友的GitHub用户名

### 标签
bug, P0
```

### 在 PR 中关联 Issue

在 PR 描述里写 `Closes #1`，合并 PR 后 Issue 会自动关闭。

## 9. 命令速查

| 想做什么 | 命令 |
|---------|------|
| 下载项目 | `git clone https://github.com/Bistu-OSSDT-2026/Verge.git` |
| 拉取最新代码 | `git pull` |
| 创建分支 | `git checkout -b feature/功能名` |
| 查看改动 | `git status` |
| 提交所有改动 | `git add -A && git commit -m "消息"` |
| 推送到 GitHub | `git push`（首次：`git push -u origin 分支名`） |
| 切换分支 | `git checkout 分支名` |
| 查看提交历史 | `git log --oneline` |
| 撤销未提交的改动 | `git checkout -- 文件名` |
| 撤销最后一次提交 | `git reset --soft HEAD~1` |
| 合并 master 到当前分支 | `git merge master` |
| 放弃所有本地改动 | `git stash` |

## 10. FAQ

### Q: 我改了代码但 Godot 里没变化？
A: 确保你在正确的分支上。用 `git branch` 查看当前分支。

### Q: `git push` 报错 "Permission denied"？
A: 你需要先加入 `Bistu-OSSDT-2026` 组织。让仓库管理员在 Settings → Collaborators 里添加你的 GitHub 账号。

### Q: 不小心在 master 分支上改了代码怎么办？
A:
```bash
git stash                  # 暂存你的改动
git checkout -b fix/临时修复  # 创建新分支
git stash pop              # 恢复改动
```

### Q: PR 合并后需要做什么？
A:
```bash
git checkout master
git pull                   # 拉取最新代码（包含你刚合并的改动）
git branch -d feature/已完成的功能名  # 删除本地分支
```

### Q: Godot 的 `.uid` 文件老是冲突怎么办？
A: `.uid` 文件已经在 `.gitignore` 中忽略了，不会冲突。如果之前已经提交了，在 Godot 中删掉这些文件重新生成即可。

### Q: AI 生成的代码能直接用吗？
A: 根据课程要求，**可以**使用 AI 辅助，但必须：
1. 实际运行验证过
2. 理解代码在做什么
3. 在 PR 描述中说明哪些部分是 AI 辅助的
